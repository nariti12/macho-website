# マチョクリッカー V2 再設計 TODO

最終更新: 2026-07-18

## 目的

マチョクリッカーを、機能数が多いだけの画面から、次の行動が直感的に分かり、クリック・購入・成長のたびに気持ちよく、短時間でも長期間でも遊び続けたくなるインクリメンタルゲームへ作り直す。

Cookie Clickerの数値や見た目をそのまま複製するのではなく、優れた中核ループ、段階的な情報開示、進行カーブ、フィードバック密度を検証可能な形で取り入れ、マチョ田の成長を主役にした独自ゲームとして完成させる。

## このTODOを作り直した理由

- 旧TODOは「機能を実装したか」を中心に完了判定しており、「面白いか」「迷わないか」「また遊びたいか」を判定できていなかった。
- 現在は初回から日課、サプリ、部位成長、筋肉結晶、設備レベル、季節イベント、実績、遺産などが同時に見え、中核ループが埋もれている。
- 建物の基礎価格と基礎CpSをCookie Clickerへ寄せても、追加倍率や報酬を含めた全体進行は同じにならない。
- 進化表示は9段階あるが、実画像は4種類しかなく、同じ画像を複数段階で再利用している。
- 旧TODOがすべてチェック済みでも、実際のプレイで「ごちゃごちゃしている」「ワクワクしない」という問題が残った。今後は自動テストだけで体験品質を完了扱いにしない。

## 調査で確認した基準

### Cookie Clicker公式実装から確認したこと

- 主資源は基本的にCookieの1種類で、クリックと自動生産が同じ資源へ集約されている。
- 建物価格は購入ごとに115%へ増加する。
- 最新版の建物は、累計Cookieがその建物の基礎価格へ到達するまで`???`と暗いアイコンで表示される。
- 未解放建物をすべて縦に見せず、先頭から連続して3件目以降の未解放行は非表示にする。常に「次に何かある」期待だけを残す。
- 購入可能、購入不可、未解放は、明度・名前・価格・アイコンの複数要素で区別される。
- Options、Stats、Legacyなどは必要なときに開く画面であり、主画面下へ常時大量表示しない。
- 大きなCookie、現在値、毎秒生産、建物、Storeの役割が画面上で明確に分離されている。

### ゲーム設計・Web仕様から採用すること

- 最初に完成させるのは「クリックする → 買えるようになる → 購入する → 毎秒生産と見た目が変わる → 次が見える」の短い中核ループとする。
- メタゲームは中核ループを補強するものだけを、必要になった時点で段階解放する。
- 音声はブラウザの自動再生制限があるため、設定上は初期ONとし、最初のクリック操作でAudioContextを開始する。
- プレイヤーへ見せる情報を増やす前に、認識負荷を減らし、次の選択肢を一つか二つへ絞る。
- 「実装済み」と「面白さ確認済み」を別の状態として扱う。

## V2の設計原則

1. 主画面は、筋肉ポイント、毎秒生産、マチョ田、設備世界、ショップだけで成立させる。
2. 新しい機能は、必要になるまで見せない。未解放の存在は`???`やシルエットで一つ先だけ予告する。
3. 購入や進化を行ったら、数値、画面、音、アニメーションの最低2種類で変化を返す。
4. 常設パネルを増やさない。詳細情報はモーダル、ドロワー、ツールチップへ移す。
5. 無料倍率を重ねて簡単にしない。倍率を追加する場合は、全体シミュレーションを更新する。
6. 画像を増やす前に、キャラクター設定、画角、透明余白、色、光源を定義する。
7. PCとスマホは別のレイアウトとして設計し、PC版を縮小しただけのスマホ画面にしない。
8. 機能は「面白さ」「判断」「次の目標」のどれにも寄与しない場合は削除する。
9. 自動テスト合格だけでは完成にしない。実機プレイとユーザー確認を完了条件に含める。
10. Cookie Clickerの著作物は流用せず、マチョクリッカー独自の画像・音・文章で作る。

## 優先度 S1.5: スマホ・PCのクリック操作を作り直す

クリック操作はゲームの中核なので、見た目の再設計より先に入力専用の仕組みとして作り直す。現在のReact `onClick`だけに依存する実装、クリックごとの複数state更新、複数`setTimeout`、音声要素の複製、無制限の粒子生成をそのまま残さない。

### 入力仕様

- [x] マウス、タッチ、ペンをPointer Eventsへ統一し、入力処理をReactコンポーネントから専用コントローラーへ分離する。
- [x] `pointerdown`で押下縮小、ハイライト、クリック音を即座に開始し、指やマウスを離すまで待たせない。
- [x] 得点加算は有効な`pointerup`で1回だけ確定し、合成`click`との二重加算を防ぐ。
- [x] キーボードのEnter/Space操作は維持し、Pointer Eventsを使わない操作でも1回だけ加算されるようにする。
- [x] `pointercancel`、画面外への移動、スクロール開始時は押下状態を必ず解除し、誤加算しない。
- [x] 押下開始位置から一定距離以上動いた操作はタップとして数えず、スマホのスクロールとクリックを区別する。
- [x] Pointer Captureを使い、指がキャラクターの輪郭から少し外れても正常に押下状態を終了できるようにする。
- [x] 右クリック、補助ボタン、想定外のペンボタンでは加算しない。
- [x] 複数指操作は許可せず、最初の`pointerId`が有効な間は2本目以降を無視する。順番に行うタップはすべて受理する。
- [x] 連打速度へ人工的な上限を設けず、描画は`requestAnimationFrame`でまとめても入力回数は落とさない。

### ダブルタップ拡大・ブラウザ誤動作の防止

- [x] マチョ田のクリック対象へ`touch-action: manipulation`を指定し、ダブルタップ拡大とそれに伴うクリック待機を無効化する。
- [x] ゲーム全体へ`user-scalable=no`を指定せず、ピンチズームによるアクセシビリティは維持する。
- [x] クリック対象へ`user-select: none`、`-webkit-user-select: none`、`-webkit-touch-callout: none`を指定し、連打中の文字・画像選択や長押しメニューを防ぐ。
- [x] `-webkit-tap-highlight-color`を調整し、ブラウザ既定の四角いタップ枠を表示しない。
- [x] 画像のドラッグ開始とコンテキストメニューを防ぎ、キャラクター画像だけが動いたり保存メニューが出たりしないようにする。
- [x] ブラウザ既定動作の抑止はマチョ田の操作領域だけへ限定し、ショップやメニューの縦スクロールを壊さない。
- [ ] iPhone Safari、Android Chromeで10秒間連打し、ページ拡大、画面スクロール、テキスト選択、長押しメニューが発生しないことを実機確認する。

### 遅延と描画負荷の削減

- [x] クリックごとの`clickBurst` stateと競合する複数`setTimeout`を廃止し、押下アニメーションをPointer EventsとCSS/Web Animations APIで直接制御する。
- [x] 得点計算は正確な整数として即時キューへ追加し、React描画は`requestAnimationFrame`単位でまとめる。
- [x] 連打中も画面上の筋肉ポイントが毎フレーム追従して増え、操作終了後にまとめて飛ぶ表示にしない。
- [x] クリック数字と粒子はオブジェクトプールを使い、上限を超えた古い要素を再利用する。
- [x] クリック音は毎回`cloneNode()`せず、事前デコードしたWeb Audioバッファまたは再利用可能な音声プールで鳴らす。
- [x] クリックハンドラー内で同期的な保存、配列全走査、画像生成、レイアウト計測を行わない。
- [x] 自動生産タイマー、背景アニメーション、ニュース更新が連打処理を妨げないよう、長いタスクと更新頻度を監査する。
- [x] クリック演出は`transform`と`opacity`を中心にし、連打ごとにレイアウト再計算を起こすプロパティを変更しない。
- [x] 軽量モードでも得点、押下縮小、数字表示、音は維持し、装飾粒子と背景演出だけを減らす。

### PCとスマホの操作感

- [x] PCでは`pointerdown`時に沈み、`pointerup`時に反発する動きとし、マウスボタンを離すまで無反応に見せない。
- [x] スマホでは指で隠れる位置を避けて獲得数字を左右へずらし、押している最中も増加を確認できるようにする。
- [x] タッチ対象はキャラクターの見た目より少し広い透明領域を持たせるが、四角い枠や背景は表示しない。
- [x] 片手操作時にショップへ誤移動せず、キャラクター中央・左右どこを押しても同じ反応を返す。
- [x] 押下、加算、音、粒子の発生順をPCとスマホで揃え、端末ごとに別ゲームのような感触にしない。
- [x] `prefers-reduced-motion`時も、押下状態と加算結果が分かる最小限のフィードバックを残す。

- [x] 新規開始時はダンベルを含む先頭の設備だけを具体名で表示し、未到達設備は`???`と暗いシルエットにする。

- [ ] 購入可能は明るさ、枠、価格色、押下感で示し、`購入可能`という重なるタグは使わない。

- [ ] 買数切替は1、10、100、最大を用意し、価格と増加量を選択数に合わせて更新する。
- [ ] ダンベル購入数はマチョ田の周囲へ順番に円周配置し、1個ずつ増えたことが見えるようにする。
- [ ] ダンベルが増える意味を、毎秒生産、設備貢献、1/10/25/50/100個の節目アップグレードで明確にする。

### S3完了条件

## 優先度 S4: 難易度を全体式で再設計する

- [ ] 購入戦略を`最安優先`、`回収時間優先`、`次設備優先`で比較する。
- [ ] 1分、5分、10分、30分、60分、4時間、24時間のポイント、CpS、所有設備、購入待ち時間を出力する。
- [ ] 最初のダンベルは約15回の基本クリックで買える基準を維持する。
- [ ] 序盤は数十秒単位で購入機会を作り、中盤以降はアップグレードや新設備が停滞を破る構成にする。
- [ ] 手動クリックが序盤で重要、中盤以降はイベントや強化時に再び重要になるバランスへする。
- [ ] ダンベルを増やすより常に上位設備だけを買う方が有利にならないよう、回収時間と節目強化を調整する。
- [ ] 進化ボーナス、実績ボーナス、シェイク、遺産を掛け合わせた最大ケースで暴走しないことを検証する。
- [ ] オフライン生産は未解放時の上限と倍率を控えめにし、長期要素で段階的に伸ばす。
- [ ] 調整係数は定数とデータへ分離し、UIコンポーネントへ直接マジックナンバーを書かない。

- [ ] どの倍率が進行を何倍にしたかをログから追跡できる。
- [ ] 5分、30分、60分の実プレイで、次の目標が遠すぎず、購入が連打だけにもならない。

## プレイテストと完成判定

### 各段階のテスト

- [ ] 5秒テスト: クリック対象と現在値が分かる。
- [ ] 5分テスト: 最初の購入と自動生産を理解し、次の目標を説明できる。
- [ ] 30分テスト: 複数設備とアップグレードを比較し、自分で購入判断ができる。
- [ ] 60分テスト: 一度は停滞し、その後の設備解放または強化で突破を感じる。
- [ ] 24時間復帰テスト: 放置報酬が進行を壊さず、次の行動を作る。
- [ ] スマホ片手テスト: 主操作、ショップ、メニューを親指で操作できる。
- [ ] 音なしテスト: 視覚だけで結果が分かる。
- [ ] 軽量モードテスト: 低性能端末でも入力遅延を感じない。

### V2完成条件

- [ ] オーナーが新規セーブで60分プレイし、現版より面白くなったと判断する。
- [ ] 説明なしテストユーザー5人中4人が、クリック、購入、自動生産、次設備解放を理解できる。
- [ ] 5人中3人以上が翌日も自発的に続きを遊ぶ。
- [ ] 初回5分の離脱理由に「情報が多い」「何をすればいいか分からない」が出ない。
- [ ] PCとスマホの両方で、重なり、切れ、矩形キャラクター背景、操作不能がない。
- [ ] Cookie Clickerとの難易度差を、全倍率込みの比較結果と意図で説明できる。
- [ ] TODOが埋まったことではなく、実プレイ結果をもって完成とする。
